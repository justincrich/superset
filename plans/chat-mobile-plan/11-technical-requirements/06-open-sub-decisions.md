# Open Technical Sub-Decisions

Deferred to sprint planning — `/kb-sprint-plan` should slot these into specific sprints:

1. **JWT lifecycle for mobile → relay**: per-call mint via cloud tRPC vs device-held longer-lived host token vs server-side proxy. Trade-off: security boundary vs latency vs offline UX.
2. **Live streaming transport**: SSE through relay (requires extending `apps/relay`'s WS tunnel to proxy `text/event-stream`) vs chunked HTTP through relay vs cloud DurableStreams SSE (existing path at `/api/chat/[sessionId]/stream`). Mobile-chat v2 may ship with periodic `chat.getSnapshot` polling and defer streaming to a follow-up mobile-chat PRD.
3. **Markdown library choice**: `react-native-markdown-display` (widely-used, opinionated styling) vs `@expensify/react-native-live-markdown` (faster but newer/less battle-tested) vs custom thin wrapper. Benchmarks needed.
4. **Tiptap WebView perf on mid-range Android**: validate that `@10play/tentap-editor` keyboard handling + input latency are acceptable on Android 11+ devices with 4GB RAM. Define a perf budget before locking in.
5. **Snapshot polling interval (if streaming deferred)**: 250ms? 500ms? 1s? Battery vs latency vs server load.
6. ~~**Host selection and workspace→host resolution**~~ — **RE-RESOLVED in v2.0.0** by the NAV functional group (see `../09-uc-nav.md`). The decision summary below supersedes the v1.6.0 host-first resolution:
   - **Project-first**, not host-first. The top-level filter axis on the sessions list is **project**, selected via a header chip (tappable bottom sheet when the active organization has ≥2 projects, static label otherwise). Host is **metadata** — rendered on individual session rows and on workspace rows in the new-chat picker, but never a global filter.
   - **`projectId` is resolved from `v2_workspaces.projectId`** (already in the schema). The mobile `chat_sessions` and `v2_workspaces` Electric collections are joined client-side; the sessions list scopes by `selectedProjectId` over the synced collection. The existing `chat_sessions` shape at `apps/electric-proxy/src/where.ts:136-137` is **unchanged** — no Electric extensions, no new APIs.
   - **`hostId` is resolved lazily from `workspace.hostId`** at chat-route mount, never at list-render time. The `useChatTunnel` hook opens the relay tunnel against that `hostId` on mount and drops it on unmount; no warm tunnels from the list. The hook de-duplicates concurrent opens to the same `hostId` across remounts so route transitions don't flap.
   - **Selected project persisted locally** in `expo-secure-store` keyed by `userId + organizationId`. First-launch default: the project whose synced sessions have the most-recent `lastActiveAt`, falling back to alphabetical-first when no sessions exist. **One-time migration** on first launch post-upgrade drops any legacy `selectedHostId` from secure-store and seeds `selectedProjectId` via the default-selection logic. Fallback when the persisted `selectedProjectId` is no longer accessible (deleted, permission revoked, org switched): re-run default selection and surface a brief toast.
   - **Offline hosts** still surface the existing UC-PLATF-03 banner — now scoped to the active chat-route (where the tunnel actually opens), not the list screen.
   - **Push-notification deep-links** (UC-NAV-05) silently align `selectedProjectId` to `workspace.projectId` before pushing the chat route; the tunnel opens lazily after the chat-route mounts. A bounded readiness gate awaits `v2_workspaces` collection sync; if the workspace row isn't synced by timeout (cold-launch race), the deep-link handler falls back to a tRPC `chat.getSnapshot({ sessionId })` fetch to resolve the workspace inline.

   **Why the change.** The v1.6.0 host-first model assumed one selected host owns the list. Workspaces in the schema span multiple hosts per project (`v2_workspaces.hostId` + `v2_workspaces.projectId`); host-first forced users to switch the global filter just to see a session living on another machine. Project-first matches the user's mental model — "I'm working in repo X" — and demotes the device dimension to metadata, paired with a filter bottom sheet (UC-NAV-08) for any narrowing the user explicitly wants. See UC-NAV-01, UC-NAV-04, UC-NAV-05, UC-NAV-08, and the README v2.0.0 changelog row for full details.

7. ~~**Mobile push-notification fanout architecture**~~ — **RESOLVED in v1.7.0** by `07-notifications.md`. Decision summary:
   - **Relay-side fanout** (`apps/relay`) — host pushes a new `push:lifecycle` outbound message on its existing tunnel WS; relay calls Expo Push API.
   - **Token storage in Upstash KV** keyed by `push:org:{orgId}:user:{userId}` as a set of `{ token, platform, deviceId, lastSeenAt }` entries.
   - **Token registration** via new relay endpoints `POST /push/register` and `DELETE /push/register/:deviceId`, gated by the same `verifyJWT` middleware used by chat tRPC.
   - **Events fanned out** are `Stop` + `PermissionRequest` only — no "agent failed" event in v2 (host emits none today).
   - **Foreground suppression** is local-only via route-aware `setNotificationHandler`; no server-side presence tracking.
   - **Permission flow** follows Expo best practices (pre-prompt → OS dialog → cold-launch re-check → deny → `Linking.openSettings()`).
   See `07-notifications.md` for the four ASCII diagrams (delivery flow, token lifecycle, permission state machine, foreground suppression).

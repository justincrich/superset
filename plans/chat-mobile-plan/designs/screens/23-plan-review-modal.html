Session · Drizzle migration plan

Migrate chat_messages → events table

5 steps · est. ~2 days for full cutover. Safe rollback path at every step.

Context

The existing chat_messages column is a jsonb[] array per row, which forces the host to deserialize the whole array on every read. For 100k-message sessions, this is a 12s cold-load. Switching to an append-only events table cuts that to ~80ms.

Plan

  1. Add chat_events table with composite PK (session_id, seq) and an index on session_id, created_at.
  2. Backfill from existing jsonb arrays in a background job. Process 5k rows per batch, 50ms between batches, single-tenant aware.
  3. Dual-write behind ff.chat_events_dual_write. Both reads and writes go to both paths.
  4. Flip reads to the new table after one release of soak (no rollback events).
  5. Drop the jsonb column in a follow-up migration once read path is stable.

Risks

  • Backfill duration — on the largest tenant (Coram), this is ~6h of writes. Run during their off-hours window (Sun 02:00–06:00 UTC).
  • Dual-write divergence — if the new path errors, do we fail the write or log-and-continue? Recommend log-and-continue with an alarm.
  • Storage spike — ~2× the message data during the dual-write window, ~14% during steady-state.

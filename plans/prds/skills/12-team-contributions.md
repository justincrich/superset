# Team Contributions

## Phase 1 — User Personas (product-manager + design-planner)

The product-manager phase produced four primary personas spanning the spectrum from skill power-users to cross-tool skeptics. design-planner contributed UX requirements for each.

### Justin — The Multi-Harness Power User (primary)
Senior engineer running 3+ coding agents per project (Claude Code for planning, Codex for refactors, Cursor for UI work). Cares about consistency: one skill, one keystroke, every harness. **Today**: rewrites the same prompt three ways or abandons skills outside Claude Code. **After**: authors once in Superset, types `/skill` anywhere, and Claude Code still gets native auto-invocation through the exporter.

### Priya — The Plugin Consumer
Mid-level engineer who installed a Claude Code plugin pack (e.g. `kb-sprint-plan`, `figma-use`) and lives in its skills daily. Cares that Superset doesn't make her re-import or maintain a parallel copy. **Today**: only sees these skills when chatting through Claude Code agent. **After**: same skills surface as `/kb-sprint-plan` in every workspace, even when she switches agents mid-session.

### Devon — The Skill Author
Solo dev who wants to capture repeated workflow knowledge ("our deploy checklist", "PR description template") as a reusable skill but doesn't want to learn the Claude plugin system or hand-edit dotfiles. Cares about a fast inline editor and immediate availability. **Today**: pastes the same boilerplate into chat or keeps a Notion doc open. **After**: opens Skills settings, writes a markdown body, hits save, types `/deploy-check` immediately in chat.

### Amir — The Cross-Tool Skeptic
Senior IC who already standardized on OpenCode / mastracode for his team. Cares that Superset doesn't lock him into Claude's skill format or break his existing setup. **Today**: avoids Superset because it feels Claude-Code-centric. **After**: configures his mastracode skills as a source root, sees them surface as slash commands, and gets a `load_skill` tool registered in mastracode automatically — no migration, no fork.

### User journeys (3 primary)
1. **First-launch onboarding** — User opens Superset, navigates to Settings → Skills, sees the 4 default sources auto-populated with their skill counts. Optional `+ Add custom source` for non-default paths. No setup required to get value.
2. **Mid-session authoring** — User mid-chat realizes they want to capture a workflow as a reusable skill. Opens `/settings/skills/new`, writes name + description + body, saves, returns to chat, types `/<name>`, body inlines. Total time: ~60 seconds.
3. **Cross-harness skill use** — User switches workspace from a Claude Code attached agent to a Cursor attached agent on the same project. The slash picker shows the same skill list in both. Typed `/skill-name` works identically; the persistent `HarnessSupportBanner` informs the user that Cursor uses the inlined-body delivery (vs Claude's native auto-invoke).

## Phase 2 — Architecture (product-manager + node-planner)

The node-planner phase delivered an adversarial architecture review that produced **six critical corrections** to the brainstormed design plus a complete technical decomposition. Full technical detail in [10-technical-requirements.md](./10-technical-requirements.md).

### Critical corrections to the original design

1. **Use `@parcel/watcher` 2.5.6, NOT chokidar.** Already in `apps/desktop/package.json` via `packages/workspace-fs/src/watch.ts`. Adding chokidar would mean two native FS-watcher modules in one Electron build (more native rebuild surface, larger bundle). Mitigation: use `@parcel/watcher.subscribe()` with a JS-rolled 200ms debounce.

2. **Plugin cache will hit Linux `inotify` defaults.** `~/.claude/plugins/cache/` can contain thousands of files; default `max_user_watches` is 8192. Mitigation: watch top two levels non-recursively, then re-scan per-plugin manually on add/change events. Document the symptom (silent missed events) in operator docs.

3. **Watcher events can race the importer transaction.** Original plan: cache invalidation fires from raw watcher event. New plan: per-source `p-queue` (concurrency=1) serializes events; cache invalidation fires AFTER the importer transaction commits.

4. **Plugin namespace race on parallel watcher events.** Two parallel events for two versions of the same plugin would race the unique constraint. Mitigation: per-plugin mutex OR `INSERT … ON CONFLICT DO UPDATE WHERE imported_at > excluded.imported_at`.

5. **Symlink loops in the Claude bundle.** ClaudeExporter must `realpath` both ends before linking; refuse if Superset's bundle path is a prefix of the resolved target.

6. **Migration ordering.** `skills` table + `settings.skillSources` column MUST go in ONE migration generated by `bunx drizzle-kit generate --name=add_skills_table_and_sources`.

### Open spike

**MastraExporter integration contract** — confirm whether `~/.mastracode/tools/*.ts` is auto-loaded, or whether the user has to register the tool in their workflow. Without this answered, MastraExporter is shovel-ready code with an unknown integration contract. UC-EXPORT-02 is tagged spike-pending in v1.

### Findings summary
- 16 system components identified across DB, Server, Main process, and Renderer layers
- 11 tRPC procedures + 1 observable subscription
- 3 new dependencies: `@parcel/watcher` (already shipping at 2.5.6), `p-queue` 8.0.1, `semver` 7.6.3
- 1 Drizzle migration combining the new `skills` table + `settings.skillSources` column
- 5 implementation phases with realistic complexity sizing (S/M/M/M/L)

### Information architecture
- New sidebar entry under "Editor & Workflow" group, positioned after Agents (skills are conceptual peers — both extend the harness layer)
- Sidebar entry: `id: "/settings/skills"`, `label: "Skills"`, icon `<HiOutlineBolt />` (matches the `ZapIcon` already used in `SkillToolCall.tsx`)
- Route hierarchy at `apps/desktop/src/renderer/routes/_authenticated/settings/skills/`:
  ```
  skills/
  ├── layout.tsx                  // splits sidebar + detail (mirrors agents v2)

  ├── $skillId/page.tsx           // /settings/skills/{id} — detail/editor

  ```

## Phase 4 — Holdout Scenarios (product-manager)

32 hidden scenarios generated across 16 use cases (2 per UC), stored at `/Users/justinrich/Projects/superset/.spec/scenarios/UC-{prefix}-{nn}/*.scenario.md`. Each scenario uses plain English BDD (deliberately NOT GIVEN/WHEN/THEN format, per the kb-prd-plan rule on reducing agent gaming) with a YAML frontmatter (`service: skills`, `feature`, `priority`, `type`).

Coverage breakdown:
- 12 happy-path scenarios validating core flows
- 8 edge-case scenarios covering boundary conditions (empty bodies, missing paths, parallel writes)
- 6 security scenarios (path traversal in source roots, symlink loops, prompt injection through SKILL.md body, loopback token theft)
- 6 error-handling scenarios (malformed frontmatter, watcher inotify limit, transaction rollback, OS dropped event)

These scenarios are intentionally hidden from the AC text so test-plan generation cannot trivially copy them. They are the validation truth-set the implementer must satisfy beyond the visible ACs.

## Synthesis (product-manager lead)

### Final scope is ONE shippable initiative
5 functional groups (SRC, SYNC, AUTH, CHAT, EXPORT) × 16 use cases. Within the 12-18 target. No split needed. Implementation phases (in [10-technical-requirements.md](./10-technical-requirements.md)) sequence from a walking skeleton through full exporters in 5 sprints with one human testing gate per sprint.

### Tensions resolved across phases
| Tension | Resolution |
|---|---|
| OpenCodeExporter scope (PM: Noop only / EM: Phase 5 alongside Mastra) | UC-EXPORT-03 covers Noop fallback for v1. Implementation Phase 5 lists OpenCodeExporter as stretch alongside Mastra; ships as v1.x patch if both spikes succeed. |
| Watcher choice | `@parcel/watcher` 2.5.6 (already shipping). Reject chokidar to avoid 2nd native module in Electron build. |
| Cache invalidation timing | After importer transaction commits, not on raw watcher event. |
| Watcher event races | Per-source `p-queue` concurrency=1. |
| Plugin parallel race | `INSERT … ON CONFLICT DO UPDATE WHERE imported_at > excluded.imported_at` + per-plugin mutex. |
| Migration ordering | Skills table + `settings.skillSources` column in ONE migration. |
| MastraExporter contract | Spike before Phase 5 implementation. UC-EXPORT-02 tagged spike-pending. |

### Specialist resolution
- **Lead**: `product-manager` (always)
- **Engineering Manager**: `node-planner` — production runtime is Node.js inside Electron main process. Bun is build/test only. Source uses `node:fs`, `node:os`, `node:path` exclusively.
- **UI Designer**: `design-planner` — selected from the available agent roster (no `ui-designer` agent type exists in this project's CLAUDE.md "Local Domain Experts" table; `design-planner` is the closest fit for transforming PRD into design artifacts).
